# ud v0.0.2 — Correctness & Coherence Plan

> Based on review of v0.0.1. The architecture held — the model and the
> `Ecosystem` contract need (almost) no changes. What needs work is
> _correctness_, _network behavior_, _safe defaults_, and making the clean
> architecture **real** instead of just shaped. Five milestones, each with hard
> exit criteria so you know exactly when it's done.
>
> Zero code here on purpose — this is the spec; the diffs come after.

---

## 0. What we're fixing (and what we're not)

v0.0.1 is a faithful build of the design: the four-question contract, the
universal model, lossless writes via `toml_edit`, parallel resolution. Those
stay. The issues, in priority order:

1. **The resolver is silently wrong** on most real manifests (partial versions
   like `"1.36"` always report _Current_).
2. **The network layer will get rate-limited** (discouraged JSON API + unbounded
   fan-out).
3. **Prereleases can leak** into "latest," contradicting the manifesto.
4. **The default mutates your file**, contradicting the manifesto and basic
   safety.
5. **The architecture is shaped but not finished** — empty modules, presentation
   in `main`, dead `span` data, repo pointing at the wrong owner.

**Invariant for this whole milestone:** the universal model (§1 of
`ARCHITECTURE.md`) and the `Ecosystem` trait stay stable. Exactly one small,
additive model change is permitted (M1) — flagged below. If anything else forces
a contract change, stop and reconsider, because that means v0.0.1's abstraction
didn't hold.

## M1 — Make it _correct_ (the resolver)

**Why.** `resolve()` builds `current_version` from the raw constraint string and
feeds it to `is_newer`, which calls `semver::Version::parse`. That parser
requires full `major.minor.patch`, so `"1.36"`, `"^1.2"`, and ranges all fail →
return `false` → the dependency reports _Current_ forever. `Scheme::satisfies`
is written but never called. Your own fixture (`tokio = "1.36"`,
`tracing = "0.1.40"`) is full of versions that currently can't flag.

**The change (described, not coded):**

- Derive a concrete **declared version** from the constraint by parsing it as a
  `VersionReq` and taking its lower bound, padding missing components with zero
  (`"1.36"` → `1.36.0`, `"^1.2"` → `1.2.0`, `">=1, <2"` → `1.0.0`). This is the
  version the manifest is "sitting on," and it's what dependi compares too.
- Compute **latest** = the maximum non-yanked version under the scheme's
  ordering (prerelease policy handled in M3).
- Verdict is **Outdated { target: latest }** iff `is_newer(declared, latest)`,
  else **Current**.
- Bring `Scheme::satisfies` to life by using it to **classify** the bump: if
  `latest` satisfies the manifest's constraint → a _compatible_ update; if not →
  a _breaking_ update. This is what powers the manifesto's "1 major behind your
  req" annotation and gives both `Scheme` methods a real job.
- Fix the sort: parse each candidate **once** into a comparable form and sort
  with a genuine total order; push unparseable versions to the bottom (or mark
  them errored) so `sort_by` never sees an inconsistent comparator.

**Permitted contract change (additive):** `Verdict::Outdated` may gain a
compatibility marker (e.g. `Outdated { target, breaking }`). Additive, justified
by the annotation above, and the only model touch allowed in v0.0.2.

**Exit criteria — done when:**

- ✅ `tokio = "1.36"` with availability containing `1.45` →
  `Outdated { target: 1.45 }`. _(The exact case broken today.)_
- ✅ `serde = "1.0.0"` with latest `1.0.219` → `Outdated { target: 1.0.219 }`.
  _(Keeps your existing integration test green; matches the manifesto.)_
- ✅ A dependency whose declared version equals the latest → `Current` (zero
  false positives).
- ✅ `"^1.2"` with latest `1.9` (same major) → `Outdated`, classified
  **compatible**; `"1"` with latest `2.3` → `Outdated`, classified **breaking**.
- ✅ Availability with all versions yanked → `Unsatisfiable`.
- ✅ A unit test mixing a garbage version string into availability does not
  panic and produces a stable order.
- ✅ Running against `tests/fixtures/large_project/Cargo.toml` flags a
  **non-trivial set** of outdated deps (not "all current"), and the known-old
  ones (`serde 1.0.0`, `tracing 0.1.40`) appear as outdated.

## M2 — Make it _survive the network_ (sparse index + bounded concurrency)

**Why.** `source()` hits `crates.io/api/v1/crates/{name}` — the endpoint
crates.io explicitly asks tools not to poll — and `pipeline.run` fires every
dependency at once via `join_all`. A 26-dep manifest = 26 simultaneous requests
at a throttled endpoint = 429s.

**The change:**

- Switch `source()` to the **sparse index** (`index.crates.io`). Mind the prefix
  bucketing: 1-char names live under `1/`, 2-char under `2/`, 3-char under
  `3/{first}/`, everything else under `{first2}/{next2}/`. The body is
  newline-delimited JSON; read `vers` and `yanked` per line. Keep the existing
  User-Agent.
- **Bound concurrency** in the pipeline (`buffer_unordered(N)` or a
  `Semaphore`), default N ≈ 8–16, ideally configurable later.
- Keep treating a missing crate / network error as a per-dependency
  `Errored`/`Unsatisfiable` — failure isolation must hold (§5).

**Exit criteria — done when:**

- ✅ `large_project` (26 deps) runs to completion with **zero rate-limit
  errors** across 5 consecutive runs.
- ✅ A synthetic 100-dependency manifest completes without 429s.
- ✅ At most N requests are in flight at once (assert via a counter/trace in a
  test).
- ✅ One nonexistent crate yields an error verdict **for that dep only**; every
  other dep still resolves.
- ✅ A unit test parses a recorded sparse-index body correctly, including a
  yanked line and a prerelease line.

## M3 — Make it _honest about prereleases_

**Why.** `source()` tags `prerelease` but `resolve()` never filters them, so a
crate whose newest release is an alpha would report `Outdated → 2.0.0-alpha`.
The manifesto says prereleases are opt-in, off by default.

**The change:**

- In the resolver's "latest" selection, **exclude prerelease versions by
  default**. Policy lives in the resolver (§4.5), not in any plugin.
- Add an opt-in flag (`--pre` / `--include-prerelease`). Keep the metadata; only
  the _selection_ changes.

**Exit criteria — done when:**

- ✅ Availability with newest `2.0.0-rc.1` and stable `1.9.0`: default run
  targets `1.9.0`; with `--pre`, targets `2.0.0-rc.1`.
- ✅ No dependency in the fixture is flagged outdated _to a prerelease_ in a
  default run.

## M4 — Make it _safe and CI-friendly_ (default to dry-run)

**Why.** Today bare `ud Cargo.toml` **rewrites the file**, and you need
`-y/--preview` to _not_ mutate — inverted and surprising. The manifesto shows
the opposite: `ud Cargo.toml` = "what's outdated?" (read-only), `--update` = the
write. A tool that edits your manifest just for _looking_ is a footgun.

**The change:**

- **Default = dry-run** (report only, file untouched). Writing requires an
  explicit `--update` / `-u`.
- Retire the confusing `-y`. Keep the `tree` subcommand.
- **Exit codes for CI:** `0` when everything is current, non-zero when outdated
  deps exist in check mode, a distinct code for hard errors. Document them.
- Update all help text to match the manifesto's verbs.

**Exit criteria — done when:**

- ✅ `ud <fixture>` with no flags leaves the file **byte-identical** (assert a
  hash before/after) and prints the report.
- ✅ `ud <fixture> --update` rewrites only outdated versions, losslessly —
  extend the existing comment-preservation write test to the full fixture.
- ✅ Exit code is non-zero when outdated deps exist in check mode, `0` when
  none; covered by a test and written in `--help`.
- ✅ `--help` text uses read-by-default / `--update`-to-write language
  consistent with `MANIFESTO.md`.

## M5 — Make the _architecture real_ (and point home)

**Why.** `report.rs` and `runtime.rs` are empty; rendering lives in `main.rs`;
`span` is computed in `read` but never used (writes look deps up by name);
`tracing` is a dependency but never initialized; clippy has easy wins; `read`
misses `[workspace.dependencies]` and `[target.*.dependencies]`; and the README
points at the wrong owner.

**The change (each small, all additive):**

- **Move rendering into `core::report`.** A reporter consumes a `Report` and
  emits human/colorized output (and a `--json` mode). `main.rs` should wire, not
  present — this is the "rendering is core, plugin-agnostic" boundary the
  architecture promised.
- **Decide `runtime.rs`:** either give it the concurrency cap + config/logging
  seam it was meant to hold (M2 work can land here), or delete the empty module.
  No dead scaffolding.
- **Decide `span`:** either _use_ it (span-targeted writes / a future
  diagnostic) or _remove_ the field until something needs it. Don't keep
  computed-but-unused data.
- **Initialize `tracing-subscriber`** behind a `--verbose` flag, or drop the
  `tracing` deps. Pick one.
- **Read `[workspace.dependencies]` and `[target.<cfg>.dependencies]`** in
  `CargoEcosystem::read`.
- **clippy clean:** derive `Default` for `Pipeline`/`Report`, `is_some_and` over
  `map_or(false, …)`, etc.
- **Point everything at `Yrrrrrf/ud`:** README badges and links, `Cargo.toml`
  `repository`/`documentation`. No `filllabs` references remain. _(This is your
  repo. 😄)_

**Exit criteria — done when:**

- ✅ `cargo clippy --all-targets -- -D warnings` is clean.
- ✅ No empty modules remain; no unused dependencies (verify with `cargo udeps`
  or by hand) — or each is justified in a comment.
- ✅ Presentation is invoked from `core::report`; `main.rs` contains only
  wiring + flag handling.
- ✅ A fixture with `[workspace.dependencies]` has those entries picked up by
  `read`.
- ✅ `grep -ri filllabs` over the repo returns nothing; README and `Cargo.toml`
  both say `Yrrrrrf/ud`.
- ✅ A one-line decision is recorded for `span` (kept-and-used, or removed) and
  for `runtime.rs` (filled, or removed).

## Decisions worth recording

```
DECISION: What does "current" mean for the verdict?
  A. The version literally written in the manifest (dependi-style freshness).
  B. The max version that satisfies the constraint (cargo-resolved drift).
CHOSEN: A.
REASON: The manifesto's own examples (serde 1.0.197 → 1.0.219) only make sense under A.
        ud's job is "is your manifest current," not "is your lockfile current."
REVISIT IF: users ask for lockfile-aware drift — that's an additive mode, not a replacement.
```

```
DECISION: How to derive a comparable "declared version" from a constraint?
  A. Parse as VersionReq, take the lower bound, pad missing components with 0.
  B. Strip operators and string-split.
CHOSEN: A.
REASON: Handles "1.36", carets, tildes, and ranges uniformly through one real parser,
        instead of accumulating the string-matching bugs we set out to avoid.
```

```
DECISION: Concurrency mechanism.
  A. futures::buffer_unordered(N).
  B. A Semaphore guarding spawned tasks.
CHOSEN: A for now.
REASON: Simplest bounded fan-out over the existing future stream; no task plumbing.
REVISIT IF: per-registry limits need different caps → a Semaphore (or per-source policy) generalizes better.
```

## Definition of done for v0.0.2

Ship when **all** of the following hold:

1. Every milestone's exit criteria are green.
2. `ud tests/fixtures/large_project` produces **correct** outdated verdicts
   (spot-checked against reality), with **no rate-limit errors**, leaving the
   file **untouched** in check mode and **losslessly updated** under `--update`.
3. `cargo test` and `cargo clippy -- -D warnings` are clean; no empty modules;
   no unused deps.
4. The universal model and `Ecosystem` trait are unchanged **except** the single
   additive `Verdict::Outdated` compatibility marker — proof the v0.0.1
   abstraction held.
5. The repo, top to bottom, says `Yrrrrrf/ud`.

## Risks & assumptions

- `[ASSUMPTION]` Freshness semantics (decision A) is the intended product. If
  you actually want lockfile-aware drift, M1's exit criteria change — confirm
  before building.
- `[ASSUMPTION]` The sparse index stays the right source; the JSON API is
  abandoned for version listing.
- `[REVISIT]` The `Verdict::Outdated` compatibility marker is the one place
  v0.0.2 touches the contract — keep it minimal; resist bolting more onto the
  verdict.
- `[REVISIT]` Validate the resolver against a genuinely odd version scheme (a
  `0.x` crate, a date-versioned one) before calling M1 done — `0.x` semantics
  (every minor is breaking) are the classic trap.

---

_Order to execute: M1 → M2 → M3 → M4 → M5. M1 makes it correct, M2 makes it
usable on a real project, and the rest make it safe, honest, and coherent. None
of it requires reopening the architecture._
