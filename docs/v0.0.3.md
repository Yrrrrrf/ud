# ud v0.0.3 — Stabilization

> Goal: turn the working v0.0.2 into a release someone can `cargo install` and
> _rely on_. Close the correctness gaps from the v0.0.2 review, make "it works"
> provable in CI, freeze the public surface, and ship it.
>
> **Scope is Cargo-only, on purpose.** A correct, installable, well-documented
> single-ecosystem tool is a real stable release. Proving the contract on a
> _second_ ecosystem is the road to 1.0 — out of scope here. (See the versioning
> decision below.)
>
> Zero code — this is the spec. Five milestones, hard exit criteria.

---

## 0. What "stable" means for this release

Not "feature-complete." Stable means: **correct on real manifests, provable
without a network, safe by default, installable from crates.io, and committed to
a public surface that won't shift under anyone building on it.** That last point
— freezing the model, the `Ecosystem` trait, _and_ the `--json` schema — is the
part that actually earns the word "stable."

Target tag: **0.1.0** (first stable, Cargo-only). `1.0.0` is reserved for after
the plugin contract is proven across a second ecosystem.

---

## M1 — Fix the writer (the one real bug)

**Why.** `write()` updates _every_ occurrence of a coordinate by name, and
`main` re-reads + rewrites the file once per outdated dependency. So a crate
that appears in two sections with different requirements gets clobbered, and a
large manifest does O(N) file I/O. The `span` field exists precisely to fix this
and isn't wired in.

**The change.**

- Make updating a **single pass**: collect all approved `(dependency, target)`
  edits, apply them to **one** in-memory `DocumentMut`, write the file **once**.
- Target edits by **span / section identity**, not bare coordinate name, so a
  coordinate appearing in multiple sections updates only the intended
  occurrence. (This is what finally puts `span` to work.)

**Exit criteria — done when:**

- ✅ A manifest with `serde = "1.0.0"` under `[dependencies]` and
  `serde = "1.0.5"` under `[dev-dependencies]`, both outdated to _different_
  targets, ends with **each section at its own correct target** — no clobbering.
- ✅ `--update` on the large fixture results in exactly **one** file read and
  **one** write, regardless of how many deps changed.
- ✅ The existing lossless guarantee holds (comments/formatting/inline tables
  preserved) — current write test stays green.
- ✅ **Idempotence:** running `--update` twice leaves the file byte-identical
  the second time.

## M2 — Make "it works" provable (test hardening)

**Why.** The strongest guarantees today are manual `just` recipes. A stable
release needs them as tests, and the suite must pass **offline**.

**The change.**

- Add an integration test over the `large_project` fixture asserting a
  non-trivial outdated set, no panics, and well-formed verdicts.
- Replace the live-network fetcher test with a **recorded/mocked** one (e.g.
  `wiremock`) so unit/CI runs never touch crates.io. Any test that must hit the
  network is `#[ignore]`d or feature-gated.
- Add the M1 duplicate-coordinate case and a check-mode "file unchanged" (hash
  before/after) test.
- Add a CI workflow: `fmt --check` + `clippy -D warnings` + `test`.

**Exit criteria — done when:**

- ✅ `cargo test` passes with **no network** (live tests are gated).
- ✅ The duplicate-coordinate test and the check-mode-unchanged test are in the
  suite and green.
- ✅ A CI run executes fmt + clippy + test and is green on a clean checkout.

## M3 — Cut the dead weight (cleanup & honesty)

**Why.** A stable release shouldn't ship scaffolding. `reqwest`'s `json` feature
is unused (you parse ndjson via `text()`), and `tracing` is initialized but has
zero call sites — `-v` currently shows nothing.

**The change.**

- Drop the unused `reqwest` `json` feature.
- Either **wire `tracing`** — a handful of `debug!`s in
  `detect`/`source`/`resolve`/`write` (the fetch URL and per-dep verdict are the
  useful ones) so `-v` is real — or **remove** `tracing`/`tracing-subscriber`
  and `-v` entirely. No half-wired observability.
- Audit for any other unused deps/features.

**Exit criteria — done when:**

- ✅ No unused dependencies or features remain (verified with `cargo machete` /
  `cargo udeps`).
- ✅ Either `ud -v <fixture>` prints meaningful trace (crate, URL, verdict),
  **or** tracing is gone and `-v` with it. Not in between.

## M4 — Release-grade UX (safe updates + human errors)

**Why.** Two gaps make the current behavior unsafe-by-default for a tool people
install: plain `--update` writes the absolute latest _including breaking jumps_,
and errors surface as raw debug dumps.

**The change.**

- **Separate compatible from breaking.** Compute both "latest compatible with
  your constraint" and "latest overall." `--update` applies **only compatible**
  bumps by default; breaking ones are reported with a clear marker but **not
  written** unless `--allow-breaking` (or `--latest`) is passed. _(This is the
  behavior change from v0.0.2 — confirm the default; see decision block.)_
- **Friendly errors:** missing file, unparseable manifest, and offline/network
  failures become single-line, actionable messages — not a `miette` stack.
- Document every flag and the exit codes (`0` current / `1` outdated / `2`
  error) in `--help` and the README.

**Exit criteria — done when:**

- ✅ Plain `ud --update` on the large fixture bumps compatible deps and **leaves
  breaking ones untouched** (e.g. `tower 0.4 → 0.5` is reported, not silently
  written); `--allow-breaking` applies them.
- ✅ Missing file / bad manifest / offline each produce a clear one-line error
  and the right non-zero exit code.
- ✅ `--help` and README document all flags and exit codes.

---

## M5 — Freeze & ship (the part that makes it "stable")

**Why.** Stability is a _promise_, not a feeling. The promise is: the public
surface won't move under you.

**The change.**

- **Freeze and document the public surface:** the universal model, the
  `Ecosystem` trait, and the `--json` output schema. State a one-line
  compatibility policy (changes to these are breaking and bump the version
  accordingly). The JSON schema counts — it's the machine API.
- **README for humans:** install (`cargo install ud`), usage, examples, the JSON
  schema, exit codes.
- **CHANGELOG** (e.g. `git-cliff`), tagged release.
- **Publish:** to crates.io; confirm docs.rs builds. Prebuilt binaries for major
  platforms (`cargo-dist`) are a nice-to-have, not a blocker.

**Exit criteria — done when:**

- ✅ `cargo install ud` on a clean machine yields a working `ud`
  (`ud --version`, `ud <Cargo.toml>` both run).
- ✅ Published to crates.io; docs.rs builds successfully.
- ✅ README covers install + usage + JSON schema + exit codes; CHANGELOG
  present.
- ✅ The model, `Ecosystem` trait, and JSON schema are documented as **stable**,
  with the compatibility policy written down.

## Decisions worth recording

```
DECISION: What version number is "stable"?
  A. 0.1.0 now (first stable, Cargo-only).
  B. 1.0.0 now.
CHOSEN: A.
REASON: 1.0 should mean "the plugin contract is proven and frozen," which needs a second
        ecosystem. 0.1.0 honestly says "stable and reliable, for Cargo, today."
REVISIT IF: never for this release; 1.0 comes after the npm/second-ecosystem proof.
```

```
DECISION: Does plain `--update` apply breaking jumps?   ← behavior change from v0.0.2, confirm.
  A. Compatible-only by default; breaking needs --allow-breaking/--latest.
  B. Latest-including-breaking by default (current v0.0.2 behavior), breaking just marked.
CHOSEN: A (recommended).
REASON: A tool people install should never silently cross a major boundary in their manifest.
        Compatible-by-default is the safe norm (cargo-edit, dependabot); breaking stays one flag away.
REVISIT IF: you'd rather keep B for the punchier "always newest" story — your call.
```

```
DECISION: How does the writer target edits?
  A. Single-pass, span/section-targeted, one write.
  B. Per-dep, by-name, re-read each time (current).
CHOSEN: A.
REASON: Fixes the duplicate-coordinate clobber and the O(N) I/O in one change, and finally uses `span`.
```

```
DECISION: Is the JSON schema part of the stable surface?
  A. Yes — freeze and document it.
  B. No — only the Rust types are stable.
CHOSEN: A.
REASON: For a CLI, `--json` *is* the public API. CI pipelines will parse it; treat it like one.
```

## Definition of done for v0.0.3 (→ 0.1.0)

Ship when **all** hold:

1. Every milestone's exit criteria are green.
2. Duplicate-coordinate is correct, updates are single-pass, breaking bumps are
   opt-in.
3. `cargo test` passes **offline**; CI (fmt + clippy + test) is green.
4. No unused deps/features; `-v` is real or absent.
5. Installable from crates.io with a README (install + usage + JSON schema +
   exit codes) and a CHANGELOG.
6. The model, `Ecosystem` trait, and JSON schema are documented as a **frozen,
   versioned public surface.**

## Risks & assumptions

- `[ASSUMPTION]` "Stable" here = stable & reliable **for Cargo**, not
  contract-proven across ecosystems. The road to 1.0 (a second ecosystem) is
  deliberately deferred — make sure that framing is on the README so 0.1.0 isn't
  oversold.
- `[REVISIT]` Freezing the JSON schema now constrains all future output — keep
  it minimal and additive-friendly (new fields fine, renames/removals breaking).
- `[ASSUMPTION]` Publishing prerequisites (crates.io token, CI secrets,
  `cargo-dist` setup) are operational chores, not code — budget a little time
  for the first publish.
- `[REVISIT]` Compatible-vs-breaking needs the "latest compatible" computation
  (max version satisfying the constraint) alongside "latest overall" — a small
  resolver addition; make sure it's covered by the M4 tests, including a `0.x`
  crate.

---

## Out of scope (this release)

A second ecosystem (npm / PyPI / Go) · vulnerability scanning · editor/LSP
integration · dynamic or sandboxed plugins · wasm build.

_Each fits the frozen contract later — none requires reopening it. That's the
whole point of freezing it now._
